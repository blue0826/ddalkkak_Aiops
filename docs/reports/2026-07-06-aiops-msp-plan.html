1. 접근안 비교

★ 추천 — A+B 하이브리드
컨트롤플레인(테넌트·라이선스·알림·AI·콘솔) 자체 개발 + 저장 계층은 오픈소스(VictoriaMetrics·Loki) 채택.
차별화(SCP 지원·멀티테넌트·Dedicated·AI)는 직접 만들고, 차별화 아닌 것은 안 만든다.
B. 오픈소스 조립
Grafana/Prometheus 등 + 글루 코드. 빠르지만 멀티테넌트 MSP 요구·라이선스 게이팅·Dedicated 경량 패키징에 구조적 한계.
C. 상용 재판매
Datadog/OpsRamp 등. SCP 미지원 · Dedicated 불가 — 요구사항 자체가 성립 안 함.

2. 배포 모델 (핵심 요구)

Mode C — Central (MSP 운영센터)
  • 멀티테넌트 컨트롤플레인 — 전 고객사 한 콘솔 관제
  • C-1 Agentless: 최소권한 API 자격증명 등록 → 중앙 pull
  • C-2 Collector: 고객 VPC 내 수집기 → outbound TLS 단방향
Mode D — Dedicated (고객 클라우드 설치)
  • 고객 SCP/AWS 계정에 전체 스택 설치 · 단일 테넌트
  • 폐쇄망 지원: 오프라인 라이선스(Ed25519) + 서명 업데이트 번들
  • 콜홈 기본 off — 고객 동의 시만
⚠ 원칙 — 단일 코드베이스 + deployment_profile + 라이선스 flag로만 분기. 빌드 포크 절대 금지 (Dedicated 버전 파편화 = 최대 유지보수 리스크).

3. AIOps 기능 계층 · 로드맵

레벨기능단계
L1룰/임계치 알림 · 통합 대시보드 · 테넌트 온보딩 · RBACP1 MVP
L2+L3통계 이상탐지(동적 베이스라인) + 알람 노이즈 억제 (한 세트 — 탐지만 넣으면 노이즈로 신뢰 상실)P2
L4LLM 어시스턴트 — 인시던트 요약·RCA 추천·월간보고서 초안 자동화P2~3
L5Runbook 자동조치 — 인간 승인 게이트 필수 (AI 추천, 사람 결정)P3
—Dedicated 패키징·에어갭 / 셀프서비스 포털·FinOpsP3 / P4
P0 (1~2주, Council 모드) — SCP v2 API 인벤토리 실측(메트릭·이벤트·로그·비용 가능/불가 표) + AWS 교차계정 PoC + 스택·이름 확정. ⚠ SCP 이벤트·비용 API 커버리지가 이 기획의 1순위 리스크 — 갭은 노드 에이전트 수집으로 우회 가능.

4. Premortem 리스크

■ 높음 SCP v2 API 커버리지 부족 → P0 실측 최우선 · 에이전트 우회로 확보
■ 높음 고객 자격증명 유출(사업 종료급) → envelope encryption·최소권한 role 분리·감사로그 5년
■ 중간 Dedicated 버전 파편화 → 단일 코드베이스 강제 · 지원 버전 N-1 정책
■ 중간 알람 노이즈로 운영팀 신뢰 상실 → L2·L3 한 세트 출시
■ 관리 폐쇄망 LLM 불가 → LLM 게이트웨이 추상화(사내 LLM 주입) + 라이선스 게이팅
■ 관리 외산(Datadog) 경쟁 → SCP+AWS 동시 지원 · 한국 MSP 실무 특화(월간보고·정기점검·에어갭)
